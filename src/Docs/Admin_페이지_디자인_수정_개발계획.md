# 참여자 관리 관리자 페이지 디자인 수정 개발계획

> **작성일:** 2026-05-06  
> **프로젝트:** JobMoa 참여자관리 시스템 v1.3.3  
> **대상:** `adminParticipantManagement.jsp` 중심 관리자 참여자 관리 화면  
> **목표 디자인:** 일반적으로 가장 많이 쓰이는 업무형 관리자 CRUD/데이터그리드 디자인  
> **상태:** 개발 전 계획 정리

---

## 1. 검토 범위

기존 문서는 관리자 페이지 10개 전체의 색상 통일과 CSS 공통화를 중심으로 작성되어 있었다. 이번 수정 계획은 사용 빈도가 가장 높은 **참여자 관리 관리자 페이지**를 먼저 표준화하고, 이후 다른 관리자 화면으로 확장할 수 있는 공통 디자인 기준을 만드는 것을 1차 목표로 한다.

### 확인한 주요 파일

| 구분 | 파일 |
|------|------|
| 계획 문서 | `src/Docs/Admin_페이지_디자인_수정_개발계획.md` |
| 참여자 관리 JSP | `src/main/webapp/WEB-INF/admin/adminParticipantManagement.jsp` |
| 참여자 관리 CSS | `src/main/webapp/css/adminCss/adminParticipantManagement_0.0.1.css` |
| 참여자 관리 JS | `src/main/webapp/js/adminJs/adminParticipantManagement_0.0.2.js` |
| 관리자 공통 GNB | `src/main/webapp/WEB-INF/tags/adminGnb.tag` |
| 기존 공통 디자인 | `src/main/webapp/css/participantCss/custom-modern_0.0.1.css` |

## 2. 현재 문제사항 정리

### 2.1 계획 범위가 너무 넓고 우선순위가 불명확함

기존 계획은 10개 관리자 페이지 전체를 동시에 수정하는 방향이다. 하지만 실제 업무에서 먼저 정리해야 할 화면은 참여자 검색, 목록 확인, 상세 조회, 엑셀 다운로드가 집중되는 참여자 관리 페이지다.

문제는 다음과 같다.

| 문제 | 영향 |
|------|------|
| 10개 페이지 동시 수정 계획 | 개발 범위가 커지고 검증 기준이 흐려짐 |
| 페이지별 색상 카테고리 분류 중심 | 실제 참여자 관리 업무 효율 개선과 직접 관련이 낮음 |
| 구현 대상 파일은 많지만 화면 사용 흐름 정의가 부족함 | 디자인은 바뀌어도 관리자 사용성이 크게 개선되지 않을 수 있음 |

### 2.2 일반적인 관리자 페이지 디자인과 다름

현재 참여자 관리 화면은 그라디언트 검색 패널, 큰 카드형 페이지 헤더, 강한 색상 배경을 사용한다. 일반적인 참여자/회원/고객 관리 관리자 화면은 장시간 반복 사용을 전제로 하므로 장식보다 정보 밀도와 스캔 속도가 중요하다.

현재 문제는 다음과 같다.

| 항목 | 현재 상태 | 일반 관리자 화면 기준 |
|------|----------|----------------------|
| 페이지 헤더 | 큰 카드와 설명문 중심 | 작은 타이틀, 총 건수, 주요 액션 중심 |
| 검색 영역 | 보라색 그라디언트 패널 | 흰색/연회색 필터 바, 접을 수 있는 고급검색 |
| 테이블 | 장식형 헤더 색상 | 조용한 헤더, 정렬/선택/상태 배지 중심 |
| 액션 버튼 | 상세/수정/삭제 텍스트 버튼 | 아이콘 버튼 또는 더보기 메뉴, 위험 액션 분리 |
| 상세 보기 | 대형 모달 | 우측 오프캔버스 또는 읽기 전용 상세 패널 |

### 2.3 참여자 관리 업무 흐름과 UI가 맞지 않음

`adminParticipantManagement_0.0.2.js`는 참여자 추가, 수정, 삭제를 사실상 제한하고 상세 조회 중심으로 동작한다. 그러나 JSP에는 참여자 추가 버튼과 수정 가능한 형태의 대형 모달 구조가 남아 있다.

정리해야 할 문제는 다음과 같다.

| 문제 | 정리 방향 |
|------|----------|
| `참여자 추가` 버튼이 있지만 클릭 시 안내만 표시 | 관리자 화면에서는 제거하거나 `상담 화면에서 등록` 링크로 변경 |
| 상세 모달 내부 입력 필드가 수정 폼처럼 보임 | 읽기 전용 상세 정보 패널로 변경 |
| 삭제/수정 정책이 JS 버전에 따라 다름 | 관리자 권한 정책을 UI에 명확히 반영 |
| 엑셀 다운로드가 현재 필터 조건을 사용하나 UI에서 명확히 드러나지 않음 | `현재 검색 결과 다운로드` 문구와 로딩 상태 표시 |

### 2.4 검색 필터 UX가 부족함

현재 필터는 필드가 많지만 초기화 버튼, 고급검색 접기, 필터 적용 상태 표시가 없다. 참여자 관리자는 반복 검색을 많이 하므로 필터 조작 비용이 낮아야 한다.

필요한 개선은 다음과 같다.

| 문제 | 개선 |
|------|------|
| 초기화 버튼 없음 | `초기화` 버튼 추가 |
| 필터가 항상 모두 노출됨 | 기본 필터와 고급 필터 분리 |
| 검색 결과 건수 표시 없음 | 테이블 상단에 `총 N명` 표시 |
| 지점 변경 시 상담사 목록은 바뀌지만 화면 피드백 없음 | 상담사 select 로딩/비활성 상태 추가 |
| 모바일에서 필터가 길게 늘어짐 | 1열 배치와 고급 필터 접기 적용 |

### 2.5 테이블이 관리자 데이터그리드 역할을 충분히 못함

현재 DataTables 기반 목록은 있지만 실무형 관리자 화면에서 흔히 기대하는 선택, 밀도 조절, 고정 액션, 상태 표시 체계가 부족하다.

| 문제 | 영향 |
|------|------|
| 행 선택 체크박스 없음 | 선택 다운로드, 선택 작업 등 확장 어려움 |
| 액션 컬럼이 텍스트 버튼으로 넓음 | 테이블 가독성 저하 |
| 상태 배지가 단순 색상만 사용 | 진행단계/마감여부 구분이 약함 |
| 테이블 래퍼가 반응형 전용 구조가 아님 | 작은 화면에서 overflow 가능 |
| `responsive: true` 설정이 있지만 DataTables Responsive 확장 CSS/JS가 JSP에 없음 | 기대한 반응형 동작이 보장되지 않음 |

### 2.6 CSS 구조가 중복되고 버전 관리가 불명확함

참여자 관리 CSS는 `.search-panel`, `.status-badge`, `.table-modern`, `.action-btn` 등을 자체 정의한다. 다른 관리자 CSS에도 `.table-modern`이 반복되어 있어 화면별 차이가 계속 생길 수 있다.

추가로 JSP는 `adminParticipantManagement_0.0.1.css`를 참조하지만 JS는 `adminParticipantManagement_0.0.2.js`를 참조한다. 디자인 수정 시 CSS/JS 버전 기준을 명확히 맞춰야 한다.

## 3. 목표 디자인 방향

참여자 관리 화면은 장식형 대시보드가 아니라 **회원/고객 관리형 관리자 데이터 화면**으로 정리한다. 가장 흔한 구성은 다음 순서다.

```text
상단 경로/타이틀
요약 지표
검색/필터 툴바
목록 액션 바
데이터그리드
우측 상세 패널
```

### 3.1 디자인 원칙

| 원칙 | 적용 방향 |
|------|----------|
| 조용한 색상 | 흰색, 회색, 남색 계열 중심에 브랜드 블루만 강조 |
| 높은 정보 밀도 | 큰 카드/그라디언트 대신 compact toolbar와 dense table 사용 |
| 반복 업무 최적화 | 검색, 초기화, 다운로드, 상세 확인을 한 화면에서 빠르게 처리 |
| 위험 액션 분리 | 삭제/수정은 권한과 정책이 명확할 때만 노출 |
| 반응형 보장 | 데스크톱은 데이터그리드, 모바일은 카드형 행 또는 수평 스크롤 |
| 공통화 | admin 전용 공통 CSS를 만들고 페이지별 CSS는 예외만 관리 |

### 3.2 컬러 시스템

기존 계획의 페이지별 강한 그라디언트는 제거한다. 참여자 관리 페이지는 업무형 관리자 화면에서 가장 많이 쓰는 중립 배경과 단일 주요 색상을 사용한다.

```css
:root {
    --admin-bg: #f6f8fb;
    --admin-surface: #ffffff;
    --admin-surface-muted: #f8fafc;
    --admin-border: #e5e7eb;
    --admin-text: #111827;
    --admin-text-muted: #6b7280;
    --admin-primary: #4361ee;
    --admin-primary-hover: #3f37c9;
    --admin-success: #0f9f6e;
    --admin-warning: #f59e0b;
    --admin-danger: #dc2626;
    --admin-info: #0284c7;
    --admin-radius: 8px;
    --admin-shadow: 0 1px 3px rgba(15, 23, 42, 0.08);
}
```

### 3.3 레이아웃 기준

| 영역 | 디자인 |
|------|--------|
| 페이지 헤더 | 카드 제거, `참여자 관리` 타이틀 + 설명 최소화 + 우측 `새로고침`, `엑셀 다운로드` |
| 요약 지표 | 전체/진행중/마감/취업/최근등록 등 4~5개 compact metric |
| 검색 필터 | 흰색 surface, 기본 필터 4개 우선 노출, 고급 필터는 접기 |
| 액션 바 | 총 건수, 선택 건수, 현재 필터 다운로드, 컬럼 표시 옵션 |
| 테이블 | checkbox, sticky header, compact row, badge, icon action |
| 상세 보기 | 우측 offcanvas 읽기 전용 패널 |
| 모바일 | 필터 1열, 테이블은 카드형 행 또는 `scrollX` 적용 |

## 4. 화면별 상세 계획

### 4.1 페이지 헤더

현재 큰 `card-modern` 헤더를 제거하고 관리자 화면에서 흔히 쓰는 얇은 page header로 변경한다.

```text
참여자 관리                              [새로고침] [현재 검색 결과 엑셀 다운로드]
참여자 검색, 상태 확인, 상세 조회를 관리합니다.
```

수정 기준은 다음과 같다.

| 항목 | 내용 |
|------|------|
| 높이 | 최소화, 상하 여백 `16px` |
| 배경 | 별도 카드 없음 |
| 액션 | 우측 정렬 |
| 설명문 | 한 줄만 유지하거나 생략 가능 |

### 4.2 요약 지표 영역

테이블을 보기 전에 현재 데이터 상태를 빠르게 파악할 수 있도록 compact metric row를 추가한다.

| 지표 | 설명 |
|------|------|
| 전체 참여자 | 현재 검색 조건 기준 전체 건수 |
| 진행중 | `closed = 0` |
| 마감 | `closed = 1` |
| 취업 | 진행단계가 취업인 참여자 |
| 최근 등록 | 최근 7일 등록자 |

API가 지표 값을 제공하지 않는 경우 1차 구현에서는 현재 로드된 목록 데이터를 기준으로 계산한다. 서버 페이지네이션이 도입되면 별도 요약 API로 분리한다.

### 4.3 검색/필터 툴바

기존 그라디언트 검색 패널을 일반 관리자 화면의 흰색 필터 바 형태로 바꾼다.

기본 노출 필터는 다음 4개로 제한한다.

| 필터 | 이유 |
|------|------|
| 구직번호 | 정확 검색 |
| 참여자명 | 가장 많이 쓰는 검색 |
| 지점 | 관리자 권한별 범위 검색 |
| 상담사 | 배정/담당자 확인 |

고급 필터는 접을 수 있게 구성한다.

| 고급 필터 | 내용 |
|----------|------|
| 진행단계 | IAP전, IAP후, 취업 |
| 등록일 시작/종료 | 기간 검색 |
| 마감여부 | 진행중, 마감 |

필수 버튼은 다음과 같다.

| 버튼 | 동작 |
|------|------|
| 검색 | 현재 필터로 목록 조회 |
| 초기화 | 모든 필터 초기화 후 전체 조회 |
| 고급검색 | 고급 필터 영역 열기/닫기 |

### 4.4 데이터그리드

참여자 관리의 핵심은 테이블이므로 일반적인 관리자 데이터그리드 패턴으로 변경한다.

| 컬럼 | 변경 방향 |
|------|----------|
| 선택 | 첫 컬럼에 checkbox 추가 |
| 구직번호 | 고정 폭, 정렬 가능 |
| 참여자명 | 강조 표시, 상세 패널 열기 링크 |
| 생년월일/성별 | compact 텍스트 |
| 지점 | badge 또는 일반 텍스트 |
| 상담사 | 이름 + 계정 표시 |
| 진행단계 | 상태 badge |
| 등록일 | 날짜 정렬 가능 |
| 마감여부 | 진행중/마감 badge |
| 액션 | `상세` 아이콘 버튼 1개 우선 |

테이블 스타일 기준은 다음과 같다.

| 항목 | 기준 |
|------|------|
| 헤더 | 연회색 배경, 진한 텍스트 |
| 행 높이 | 약 `44px`~`52px` |
| hover | 연한 blue/gray 배경 |
| border | 행 구분선만 사용 |
| 액션 | 아이콘 버튼 + tooltip |
| 반응형 | `.table-responsive`와 DataTables `scrollX` 우선 적용 |

### 4.5 상세 조회 패널

현재 대형 모달은 수정 폼처럼 보이기 때문에 읽기 전용 상세 확인 목적과 맞지 않는다. 일반 관리자 화면에서 많이 쓰는 우측 offcanvas 패턴으로 변경한다.

```text
오른쪽 패널
┌────────────────────┐
│ 참여자 상세         │
│ 이름 / 구직번호      │
├────────────────────┤
│ 기본 정보           │
│ 상담 정보           │
│ 희망 정보           │
│ 메모                │
├────────────────────┤
│ [상담 화면에서 수정] │
└────────────────────┘
```

정책은 다음과 같이 정리한다.

| 항목 | 정책 |
|------|------|
| 관리자 화면 기본 기능 | 조회 전용 |
| 수정 | 기존 상담 화면으로 이동하거나 권한 확정 후 별도 구현 |
| 삭제 | 기본 노출하지 않음. 필요 시 권한 있는 사용자에게만 더보기 메뉴로 노출 |
| 상세 패널 | 필드 그룹별 읽기 전용 description list 사용 |

### 4.6 엑셀 다운로드

## 6. 구현 순서

### 1단계: 공통 관리자 디자인 토큰 생성

1. `adminCommon_0.0.1.css`를 생성한다.
2. `:root`에 관리자 전용 색상, 간격, radius, shadow 변수를 정의한다.
3. 버튼, badge, table, filter, loading overlay 공통 클래스를 작성한다.
4. 기존 `custom-modern_0.0.1.css`와 충돌하지 않도록 `.admin-*` prefix만 사용한다.

### 2단계: 참여자 관리 JSP 구조 변경

1. `adminParticipantManagement.jsp`에 `adminCommon_0.0.1.css`를 추가한다.
2. 기존 `adminParticipantManagement_0.0.1.css` 참조를 `adminParticipantManagement_0.0.2.css`로 변경한다.
3. JS 참조를 `adminParticipantManagement_0.0.3.js`로 변경한다.
4. 큰 카드형 페이지 헤더를 `.admin-page-header`로 교체한다.
5. 요약 지표 영역을 추가한다.
6. 검색 패널을 `.admin-filter-panel` 기반으로 재구성한다.
7. 초기화 버튼과 고급검색 토글 버튼을 추가한다.
8. 테이블 첫 컬럼에 선택 checkbox를 추가한다.
9. 기존 대형 모달을 Bootstrap offcanvas 읽기 전용 상세 패널로 교체한다.

### 3단계: 참여자 관리 전용 CSS 작성

1. `adminParticipantManagement_0.0.2.css`를 생성한다.
2. 참여자 관리 테이블 컬럼 폭을 정의한다.
3. 상세 offcanvas 내부의 field group, label, value 스타일을 정의한다.
4. 모바일에서 metric/filter/table이 깨지지 않도록 media query를 추가한다.
5. 기존 `.search-panel`, `.table-modern`, `.status-badge`, `.action-btn` 의존성을 제거한다.

### 4단계: 참여자 관리 JS 정리

1. `adminParticipantManagement_0.0.3.js`를 생성한다.
2. DataTables 초기화에 `scrollX`, `autoWidth: false`, `pageLength: 25`를 명시한다.
3. `responsive: true`는 Responsive 확장을 추가하지 않을 경우 제거한다.
4. 검색 파라미터 생성 함수를 `getParticipantSearchParams()`로 분리한다.
5. `resetParticipantFilters()`를 추가한다.
6. 목록 로드 성공 시 metric 값을 계산해 갱신한다.
7. 상세 버튼 클릭 시 offcanvas를 열고 읽기 전용 데이터를 채운다.
8. 엑셀 다운로드는 현재 필터 조건을 사용하고 공통 로딩 overlay를 표시한다.

## 9. 완료 기준

### 9.1 사용성

- [ ] 참여자 검색에 필요한 주요 필터가 첫 화면에서 보인다.
- [ ] 검색 결과 건수와 상태 요약을 즉시 확인할 수 있다.
- [ ] 현재 검색 조건으로 엑셀 다운로드가 가능하다는 점이 명확하다.
- [ ] 상세 조회가 수정 폼처럼 보이지 않는다.
- [ ] 관리자 화면에서 지원하지 않는 추가/수정/삭제 버튼이 혼란을 주지 않는다.

### 9.2 디자인

- [ ] 강한 그라디언트 검색 패널이 제거된다.
- [ ] 흰색/회색 surface와 단일 primary color 중심의 업무형 UI가 적용된다.
- [ ] 테이블, badge, 버튼, 필터 간격이 공통 CSS 기준을 따른다.
- [ ] 텍스트와 버튼이 모바일에서 겹치거나 잘리지 않는다.

### 9.3 기술

- [ ] JSP/CSS/JS 버전 참조가 일관된다.
- [ ] DataTables 반응형 설정과 실제 로드된 플러그인이 일치한다.
- [ ] 기존 API URL과 파라미터는 유지된다.
- [ ] 새 CSS는 `.admin-*` prefix를 사용해 기존 참여자 페이지 CSS와 충돌하지 않는다.
- [ ] Chrome DevTools 기준 모바일/태블릿/데스크톱 확인이 완료된다.

## 10. 이번 계획의 확정 방향

1. 10개 관리자 페이지 전체 개편보다 **참여자 관리 페이지 우선 개편**으로 범위를 줄인다.
2. 기존 장식형 그라디언트 디자인 대신 **일반 관리자 CRUD/데이터그리드 디자인**을 적용한다.
3. 참여자 관리 화면은 **검색/요약/목록/상세 조회/엑셀 다운로드** 중심으로 재정리한다.
4. 추가/수정/삭제처럼 현재 정책상 제한된 기능은 화면에서 혼란을 주지 않도록 제거하거나 권한 조건으로 숨긴다.
5. 공통 CSS는 만들되, 첫 적용 대상은 참여자 관리 화면으로 제한한다.
6. 참여자 관리 화면 안정화 후 같은 `.admin-*` 공통 컴포넌트를 다른 관리자 페이지로 확장한다.
