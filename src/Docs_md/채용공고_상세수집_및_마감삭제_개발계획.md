## 2. 변경사항 1 — 마감 공고 삭제 처리

### 2-1. 현재 문제
- `deactivateOldPostings()` 가 `is_active = 0` 으로 UPDATE만 하므로 마감된 공고가 테이블에 계속 쌓임
- `searchWhere` 쿼리의 `WHERE is_active = 1` 조건이 존재해 검색 시 전체 테이블 스캔 가능성 있음

### 2-2. 변경 내용

**Recruitment-mapping.xml**
```xml
<!-- 기존 -->
<update id="deactivateOldPostings">
    UPDATE J_참여자관리_JOB_POSTING
    SET    is_active = 0
    WHERE  sync_dtm  < #{syncDtm}
      AND  is_active = 1
</update>

<!-- 변경 후 -->
<delete id="deleteOldPostings" parameterType="String">
    DELETE FROM J_참여자관리_JOB_POSTING
    WHERE sync_dtm &lt; #{syncDtm}
</delete>
```

- `searchWhere` 의 `WHERE is_active = 1` 조건 **제거** (삭제 처리 후 테이블에는 활성 데이터만 존재)
- `upsertPosting` MERGE 문에서 `is_active` 컬럼 SET/INSERT 구문 **제거**

**RecruitmentDAO.java**
```java
// 기존
public int deactivateOldPostings(String syncDtm) { ... }

// 변경 후
public int deleteOldPostings(String syncDtm) {
    return sqlSession.delete(NS + "deleteOldPostings", syncDtm);
}
```

**RecruitmentScheduler.java**
```java
// 기존
int deactivated = recruitmentDAO.deactivateOldPostings(syncDtm);
log.info("[Scheduler] 마감 공고 비활성화: {}건", deactivated);

// 변경 후
int deleted = recruitmentDAO.deleteOldPostings(syncDtm);
log.info("[Scheduler] 마감 공고 삭제: {}건", deleted);
```

**RecruitmentPostingDTO.java**
- `isActive` 필드 제거 (또는 유지하되 DB 저장에서 제외)

### 2-3. DB 컬럼 처리
- `J_참여자관리_JOB_POSTING` 테이블의 `is_active` 컬럼은 **즉시 삭제하지 않음**
- 운영 배포 후 일정 기간(1~2주) 정상 동작 확인 뒤 DDL로 컬럼 제거 예정
- 컬럼 제거 전까지 UPSERT에서 `is_active` 값을 고정 `1`로 넣거나 SET/INSERT 구문에서 제외

## 3. 변경사항 2 — 신규 공고 상세정보 수집 (B 방식)

### 3-1. 설계 원칙

| 항목 | 결정 |
|---|---|
| 수집 대상 | **신규 INSERT된 공고만** (기존 UPDATE 공고 제외) |
| 실행 시점 | 메인 동기화 스케줄러 완료 직후 연속 실행 |
| API 호출 전략 | 100건 단위 배치 처리 + 건당 딜레이(100ms) |
| 상세 데이터 저장 | 기존 `JOB_POSTING` 테이블에 컬럼 추가 (별도 테이블 없음) |
| 신규 식별 방법 | `detail_fetched` 컬럼 (BIT, 기본값 0) 활용 |

### 3-2. 고용24 상세 API

```
URL: https://www.work24.go.kr/cm/openApi/call/wk/callOpenApiSvcInfo210D01.do
파라미터:
  - authKey    : API 인증키
  - callTp     : D
  - returnType : XML
  - wantedAuthNo : 구인인증번호 (공고 PK)
  - infoSvc    : VALIDATION (필수: 워크넷 제공처 지정)
```

> 상세 API URL 및 응답 필드는 고용24 API 명세서에서 최종 확인 필요

### 3-4. UPSERT 변경 — INSERT 시 `detail_fetched = 0` 설정

```xml
WHEN NOT MATCHED THEN
    INSERT (
        ..., detail_fetched
    ) VALUES (
        ..., 0   -- 신규 공고는 미수집 상태로 INSERT
    );
-- WHEN MATCHED (UPDATE) 시에는 detail_fetched 컬럼을 건드리지 않음
-- (이미 상세정보를 수집한 기존 공고 정보 보존)
```

```
        │     └── updateDetailFetched()  — JOB_POSTING 상세 컬럼 UPDATE + detail_fetched=1
        │
        └── 100건마다 Thread.sleep(1000ms) 딜레이
```

### 3-7. RecruitmentServiceImpl 상세 수집 메서드 (예시)

```java
/**
 * detail_fetched=0 인 신규 공고 상세정보 수집
 * 최대 500건 제한 (1회 스케줄 실행 기준)
 */
public void fetchDetailForNewPostings() {
    List<String> newIds = recruitmentDAO.selectNewPostingIds(500);
    if (newIds.isEmpty()) {
        log.info("[상세수집] 신규 공고 없음 — 상세 수집 생략");
        return;
    }
    log.info("[상세수집] 신규 공고 {}건 상세 수집 시작", newIds.size());

    int success = 0, fail = 0;
    for (int i = 0; i < newIds.size(); i++) {
        String wantedAuthNo = newIds.get(i);
        try {
            RecruitmentDetailDTO detail = fetchDetail(wantedAuthNo);
            recruitmentDAO.updateDetailFetched(detail);
            success++;
        } catch (Exception e) {
            log.warn("[상세수집] 실패 wantedAuthNo={}: {}", wantedAuthNo, e.getMessage());
            fail++;
        }
        // 100건 처리마다 1초 대기 (고용24 API Rate Limit 방지)
        if ((i + 1) % 100 == 0) {
            try { Thread.sleep(1000); } catch (InterruptedException ignored) {}
        }
    }
    log.info("[상세수집] 완료 — 성공: {}건, 실패: {}건", success, fail);
}
```

## 5. 고려 사항 및 주의점

| 항목 | 내용 |
|---|---|
| 상세 수집 최대 건수 | 1회 500건 제한 → 미수집 건이 쌓이면 다음 스케줄에 자동 처리 |
| 상세 API 실패 처리 | 실패 시 `detail_fetched = 0` 유지 → 다음 동기화 때 재시도 자동화 |
| 마감 DELETE 순서 | UPSERT 완료 후 DELETE 실행 (순서 바뀌면 신규 공고도 삭제될 위험) |
| `is_active` 컬럼 | 배포 후 2주 안정 확인 뒤 DDL로 컬럼 제거 예정 |
| 고용24 Rate Limit | 100건당 1초 딜레이 기본 적용, 429 응답 시 딜레이 증가 검토 |
