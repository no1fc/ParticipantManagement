# AI_Function_PRD.md — 참여자관리 검색/리스트/AI 추천 기능 고도화

## 1. 개요

본 문서는 참여자관리 화면의 검색 기능 개편, 리스트 UI 확장, AI 기반 채용정보 추천 기능 추가, 그리고 이와 연계된 백엔드/DB/화면 연동 작업의 요구사항과 개발 범위를 정의한다.

해당 기능은 상담사가 사용할 것을 기준으로 설계하며, 향후 카카오톡 공유 기능 확장을 고려한 구조로 개발한다.

---

## 2. 배경 및 목적

현재 참여자관리 기능은 기본적인 조회 중심으로 동작하고 있으나, 상담사가 실제 업무에서 활용하기에는 아래 기능이 부족하다.

- 특정 조건의 참여자 검색이 어렵다
- 집중 알선 대상자를 별도로 분류하기 어렵다
- 참여자별 추천 채용정보를 한 화면에서 확인하기 어렵다
- 참여자 데이터와 채용정보를 기반으로 한 추천 기능이 없다
- 이후 외부 공유(카카오톡) 확장성을 고려한 데이터 구조가 부족하다

따라서 본 프로젝트는 다음 목적을 가진다.

1. 참여자 검색 정확도 및 편의성 개선
2. 상담 업무용 리스트/모달 기능 강화
3. AI 기반 추천 채용정보 기능 도입
4. 향후 공유 기능까지 고려한 확장 가능한 설계 확보

---

## 3. 목표

### 3.1 업무 목표
- 상담사가 참여자를 빠르게 검색하고 분류할 수 있어야 한다.
- 집중 알선 대상 참여자를 별도로 필터링할 수 있어야 한다.
- 참여자 상세 정보를 기반으로 추천 채용정보를 즉시 확인할 수 있어야 한다.
- AI 추천 결과를 통해 상담사의 채용 연계 업무를 보조해야 한다.

### 3.2 기술 목표
- 기존 DB 구조 및 쿼리를 최소한으로 침범하면서 확장 가능하게 변경한다.
- 참여자관리 검색 조건과 리스트 UI를 분리 가능한 구조로 정리한다.
- 추천 기능은 추후 외부 공유 기능과 연결될 수 있도록 API/데이터 응답 구조를 표준화한다.

## 4. 사용자 및 사용 시나리오

### 4.1 사용자
- 상담사
  - 참여자 검색
  - 집중 알선 여부 확인
  - 추천 채용정보 확인
  - 추후 카카오톡 공유 예정

### 4.2 주요 시나리오
1. 상담사가 참여자관리 화면에서 이름, 조건, 상태 등을 입력해 검색한다.
2. 집중 알선 대상 참여자만 별도로 필터링한다.
3. 검색 결과 리스트에서 특정 참여자의 추천 채용정보 버튼을 클릭한다.
4. 모달에서 참여자 정보와 추천 채용정보 리스트를 확인한다.
5. 이후 공유 기능이 추가되면 추천 결과를 카카오톡으로 전송한다.

## 6. 기능 요구사항

### 6.1 참여자관리 검색 기능 수정

#### 6.1.1 DB 수정
- 집중 알선 여부를 저장할 수 있는 컬럼을 추가한다.
- 컬럼명은 프로젝트 네이밍 규칙에 맞게 정의한다.
- 해당 값은 참여자별 검색 필터 및 추천 대상 구분에 활용한다.

#### 6.1.2 BackEnd DB 쿼리 수정 및 연동
- 기존 참여자 검색 쿼리를 수정하여 신규 컬럼을 반영한다.
- 집중 알선 여부 조건이 검색에 포함될 수 있도록 한다.
- 검색 조건에 따라 목록이 정확히 필터링되어야 한다.

#### 6.1.3 검색 필터 수정
- 집중 알선 참여자를 검색할 수 있는 필터를 추가한다.
- 기존 검색 조건과 함께 조합 검색이 가능해야 한다.
- 상담사 업무 흐름상 자주 쓰는 조건은 기본 UI에서 바로 접근 가능해야 한다.

---

### 6.2 참여자관리 리스트 내용 수정

#### 6.2.1 리스트에 버튼 추가
- 참여자 리스트에 “참여자 알선 추천 채용정보 모달 창” 버튼을 추가한다.
- 버튼은 각 참여자 행(row) 기준으로 노출한다.
- 버튼 클릭 시 해당 참여자의 추천 정보 모달을 띄운다.

#### 6.2.2 참여자 알선 추천 모달
모달은 다음 정보를 포함한다.

##### 6.2.2.1 참여자 알선 정보 출력 디자인
- 참여자 기본 정보
- 상담/알선 판단에 필요한 핵심 정보
- 집중 알선 여부
- 추천에 참고되는 주요 상태값

##### 6.2.2.2 참여자 알선 추천 채용정보 리스트 출력 디자인
- 추천 채용정보 리스트
- 채용정보 제목
- 회사명
- 근무조건 핵심 요약
- 추천 근거 또는 매칭 포인트
- 필요 시 상세보기/공유 버튼 확장 가능 구조

### 6.3 AI 기능 추가

#### 6.3.1 기능 개요
- 고용24 채용정보 DB와 참여자 데이터를 기반으로 추천 채용정보를 산출한다.
- 추천 결과는 상담사가 즉시 활용할 수 있어야 한다.
- 추천은 단순 조회가 아니라 참여자 특성과 채용 조건을 함께 반영해야 한다.

#### 6.3.2 데이터 소스
- 채용정보
  - 고용24 채용정보 DB
  - 스케줄러를 통해 00시, 06시, 12시, 18시에 적재됨
- 참여자 정보
  - 상담사가 저장한 참여자 데이터
  - 현재 정보 조회 기준은 학력, 전공, 카테고리 대/중 분류, 희망직무, 알선상세정보를 포함한다.

#### 6.3.3 AI 추천 방향성
이번 기능은 모든 참여자에게 동일한 AI 추천을 적용하지 않는다.  
참여자의 진행단계와 알선상세정보 유무에 따라 추천 방식이 달라진다.

##### 적용 원칙
- **대상 참여자**
  - 진행단계가 `IAP후`, `미취업사후관리`인 경우에만 추천 버튼을 통해 AI 추천 저장이 가능하다.
  - 이 범위 외 참여자는 추천 저장 대상에서 제외한다.

- **참여자 정보 기준**
  - 학력
  - 전공
  - 카테고리 대분류
  - 카테고리 중분류
  - 희망직무
  - 알선상세정보(상담사 입력)

##### 추천 처리 방식
1. **기본 검색 조건 생성**
  - 참여자의 기본정보를 기반으로 Gemini API를 활용하여 고용24 채용정보 DB 검색 조건을 생성한다.
  - 검색 조건은 채용공고 후보군을 좁히는 데 사용한다.
  - 검색 조건 생성 시 기준 필드는 학력, 전공, 카테고리 대/중 분류, 희망직무이다.

2. **알선상세정보가 있는 참여자**
  - 기본 검색 조건으로 후보군을 조회한 뒤,
  - 해당 후보군과 참여자 알선상세정보를 다시 Gemini API에 전달한다.
  - AI가 후보군 중 가장 적합한 채용정보를 선별하고,
  - 베스트 채용정보와 검색 리스트를 함께 저장한다.

3. **알선상세정보가 없는 일반 참여자**
  - 기본 검색 조건만 만족하면 해당 시트를 저장한다.
  - 이 경우 AI 재판단 단계는 생략한다.
  - 즉, 후보군 검색 결과 중심으로 추천 리스트를 저장한다.

##### 설계 의도
- Gemini API는 모든 상황에서 직접 추천을 생성하는 도구가 아니라,  
  **검색 조건 생성 및 2차 정밀 판단 보조 도구**로 사용한다.
- 상담사 입력값이 풍부한 경우에는 더 정밀한 추천 결과를 저장한다.
- 상담사 입력값이 부족한 경우에도 최소한의 추천 후보군은 저장한다.
- 이를 통해 토큰 비용과 응답 지연을 줄이면서도 실무 활용도를 유지한다.

#### 6.3.4 추천 저장 DB 설계
- DB명: `J_참여자관리_참여자추천채용정보`
- SQL Server(MSSQL) 기준으로 설계한다.
- 컬럼명은 한국어로 작성 가능하다.
- 저장 목적은 추천 결과의 조회, 재사용, 추후 공유 확장이다.

권장 저장 항목 예시:
- 구직번호
- 참여자명
- 진행단계
- 학력
- 전공
- 카테고리대분류
- 카테고리중분류
- 희망직무
- 알선상세정보
- 생성된검색조건
- 추천채용정보_URL
- 추천채용정보_기업명
- 추천채용정보_제목
- 추천채용정보_업종
- 베스트채용정보
- 추천점수
- 추천사유
- 저장일시
- 수정일시

#### 6.3.5 저장/갱신 규칙
- 추천 버튼 클릭 시 저장 또는 갱신이 가능해야 한다.
- 동일 참여자에 대해 중복 저장이 발생하지 않도록 키 규칙을 정의한다.
- 신규 채용공고가 유입되었을 때는 기존 참여자 추천 결과를 재검토할 수 있도록 설계한다.
- 향후 추천 결과가 카카오톡 공유와 연결될 수 있도록 조회용 응답 구조를 분리한다.

#### 6.3.6 추천 결과 응답 구조 요구
- 참여자 기본 정보
- 생성된 검색 조건
- 추천 채용정보 리스트
- 베스트 채용정보
- 추천 근거 항목
- 공유용 요약 메시지 초안

### 6.4 화면, 기능 연동 작업
- 검색 화면, 리스트 화면, 모달 화면, 백엔드 API, DB 쿼리를 상호 연동한다.
- 화면 단에서 필요한 검색 조건과 백엔드 파라미터를 일치시킨다.
- 추천 모달은 목록 조회와 독립적으로 동작할 수 있어야 한다.
- 화면 전환 없이 모달 중심 UX로 구현하여 상담사 사용성을 높인다.
- 추천 저장 버튼은 진행단계 조건에 맞는 경우에만 활성화한다.

## 7. 비기능 요구사항

### 7.1 성능
- 검색 결과는 상담 업무에 지장이 없도록 빠르게 반환되어야 한다.
- 추천 모달 호출 시 체감 지연을 최소화해야 한다.
- 채용정보가 많더라도 페이지 응답이 과도하게 느려지지 않아야 한다.

### 7.2 유지보수성
- 검색 조건, 추천 로직, UI는 분리 가능한 구조로 설계한다.
- 추천 기준이 바뀌어도 전체 화면을 크게 수정하지 않도록 한다.
- 추후 카카오톡 공유 기능 추가 시 재사용 가능한 API 구조를 유지한다.

### 7.3 확장성
- 추천 로직은 향후 AI 모델 교체가 가능해야 한다.
- 공유 기능은 추천 결과 응답을 그대로 활용할 수 있어야 한다.

### 7.4 안정성
- 추천 기능 실패 시에도 기본 검색/조회 기능은 정상 동작해야 한다.
- Gemini API 실패 시 대체 메시지 또는 빈 추천 상태를 노출한다.
- 저장 실패 시 중복 데이터가 쌓이지 않도록 예외 처리를 둔다.

## 9. 성공 기준

다음 조건을 만족하면 기능 목표를 달성한 것으로 본다.

1. 집중 알선 참여자를 검색할 수 있다.
2. 참여자 리스트에서 추천 채용정보 모달을 열 수 있다.
3. 모달에서 참여자 정보와 추천 채용정보가 정상 표시된다.
4. Gemini API 기반 검색 조건 생성이 동작한다.
5. 알선상세정보 유무에 따라 저장 방식이 다르게 처리된다.
6. 진행단계가 `IAP후`, `미취업사후관리`인 경우에만 AI 추천 저장 버튼이 동작한다.
7. 향후 카카오톡 공유 기능을 무리 없이 추가할 수 있는 구조를 확보한다.

## 10. 리스크 및 대응

### 10.1 참여자/채용정보 데이터 정합성 문제
- 리스크: 데이터 필드가 불완전하거나 값 형식이 다를 수 있음
- 대응: 매칭 전에 정규화 및 예외처리 필요

### 10.2 Gemini API 비용 및 호출량 증가
- 리스크: 호출이 잦아질 경우 비용 증가 가능
- 대응: 저장형 추천 구조를 기본으로 하고, 불필요한 재호출을 줄인다.

### 10.3 추천 품질 부족
- 리스크: 초기 추천이 실무 기대에 못 미칠 수 있음
- 대응: 검색 조건 생성과 최종 판정 단계를 분리하고, 추천 사유를 함께 저장한다.

### 10.4 응답 지연
- 리스크: 참여자/채용정보 조합 계산으로 느려질 수 있음
- 대응: 점수 계산 범위 제한, 후보군 축소, 저장 결과 재사용 검토

### 10.5 추후 공유 기능 연동 난이도
- 리스크: 추천 데이터 구조가 공유 기능과 맞지 않을 수 있음
- 대응: 공유용 응답 포맷을 초기에 같이 설계

## 12. 개발 일정표
대상: 2~3년차 주니어 개발자 1명 기준, 현실적인 단독 개발 일정

> 전제:
> - 기존 프로젝트 구조 파악 및 수정 병행
> - 업무 중간에 피드백/수정 반영 시간 포함
> - 하루 6~7시간 실개발 기준
> - 전체 일정은 여유 포함 약 8~12주 권장

### 12.1 일정 개요

| 주차 | 작업 내용 | 산출물 |
|---|---|---|
| 1주차 | 요구사항 정리, 기존 참여자관리/채용정보 흐름 분석, DB 구조 확인 | 작업 범위 확정서, 영향 범위 메모 |
| 2주차 | DB 수정 설계, 검색 조건 정의, 참여자 테이블/쿼리 영향 검토 | DB 변경안, 쿼리 변경안 |
| 3주차 | 백엔드 검색 쿼리 수정, 필터 파라미터 연동 | 검색 API/쿼리 1차 완료 |
| 4주차 | 참여자관리 화면 리스트 수정, 버튼 추가, 모달 기본 레이아웃 | UI 기본 구현 |
| 5주차 | 참여자 정보/추천 채용정보 모달 연동, 리스트 출력 | 모달 데이터 연동 완료 |
| 6주차 | Gemini API 연동 및 검색 조건 생성, 추천 저장 로직 1차 구현 | 추천 저장 API/서비스 |
| 7주차 | 추천 결과 사유 출력, 예외 처리, 성능 보완 | 추천 품질 개선 |
| 8주차 | 전체 화면/기능 통합 테스트, 버그 수정 | 통합 테스트 완료 |
| 9주차 | 상담사 피드백 반영, 디자인/문구 수정 | 사용성 개선 완료 |
| 10주차 | 카카오톡 공유 대비 응답 구조 정리, 확장 포인트 분리 | 공유 확장 설계안 |
| 11~12주차 | 최종 안정화, 운영 반영, 문서화 | 배포 가능 상태, PRD/운영 문서 |

### 12.2 현실적인 개발 우선순위 일정

#### 1단계: 필수 기능 안정화
- 검색 필터
- DB 컬럼 추가
- 리스트 버튼
- 모달 조회
- 추천 결과 저장

#### 2단계: AI 연동
- Gemini API 검색 조건 생성
- 알선상세정보 유무에 따른 분기
- 추천 사유 및 베스트 채용정보 저장

#### 3단계: 확장 준비
- 카카오톡 공유용 구조 분리
- 추천 로그/이력/재사용 API 정리

### 12.3 예상 총 소요
- 최소 일정: 6~8주
- 권장 일정: 8~12주
- 여유 포함 안정 일정: 3개월 내외

혼자 진행하는 경우, 기능 구현보다 기존 구조 파악, 테스트, 피드백 반영 시간이 더 많이 든다.  
따라서 위 일정은 현실적으로 잡은 편이다.
