# 채용공고 스케줄러 OOM 대응 및 운영 점검

작성일: 2026-05-20

## 1. 배경

운영 Ubuntu 서버에서 채용공고 동기화 스케줄러 실행 후 Tomcat JVM에 아래 오류가 발생했다.

```text
java.lang.OutOfMemoryError: Java heap space
Exception in thread "http-nio-8080-Acceptor" java.lang.OutOfMemoryError: Java heap space
```

당시 로그 흐름:

```text
12:00:00 [Scheduler] 채용공고 동기화 시작
12:00:02 동기화 대상 총 57655건, 예상 577페이지
12:08:11 [Scheduler] API 수집 완료: 57677건
13:28:17 [Scheduler] DB UPSERT 완료: 57677건
13:28:17 [상세수집] 신규 공고 1228건 상세 수집 시작
13:32:40 [Scheduler] 채용공고 동기화 완료
이후 OutOfMemoryError 발생
```

원인은 Ubuntu 서버 자체 종료가 아니라, Tomcat 내부 JVM heap 부족으로 판단한다.

## 2. 기존 문제

기존 스케줄러 흐름:

```text
고용24 API 전체 페이지 호출
→ 전체 결과를 List<RecruitmentPostingDTO>에 누적
→ 약 5만 건 이상을 한 번에 DB UPSERT
→ 신규 공고 상세수집 연속 실행
```

이 방식은 XML 응답 문자열, DTO 리스트, DB 저장 대상 객체가 한 번에 JVM 메모리에 남기 때문에 채용공고 수가 많을 때 OOM 가능성이 높다.

## 3. 수정 내용

### 3.1 페이지 단위 저장

수정 후 스케줄러 흐름:

```text
1페이지 API 호출
→ 100건 즉시 UPSERT
→ 2페이지 API 호출
→ 100건 즉시 UPSERT
→ 반복
```

전체 채용공고를 JVM 메모리에 누적하지 않는다.

관련 파일:

| 파일 | 내용 |
|------|------|
| `src/main/java/com/jobmoa/app/recruitmentFormation/biz/RecruitmentScheduler.java` | 스케줄러가 `syncPostingsByPage(syncDtm)` 호출 |
| `src/main/java/com/jobmoa/app/recruitmentFormation/biz/RecruitmentService.java` | 페이지 단위 동기화 메서드 추가 |
| `src/main/java/com/jobmoa/app/recruitmentFormation/biz/RecruitmentServiceImpl.java` | 페이지 조회/저장/실패 처리 구현 |
| `src/main/java/com/jobmoa/app/recruitmentFormation/biz/dto/RecruitmentSyncResultDTO.java` | 동기화 결과 DTO 추가 |
| `src/test/java/com/jobmoa/app/recruitmentFormation/biz/RecruitmentServiceImplPageSyncTest.java` | 페이지 단위 저장 테스트 추가 |

### 3.2 실패 시 기존 데이터 보호

아래 상황이 발생하면 기존 공고 삭제와 상세수집을 생략한다.

- API 페이지 조회 실패
- API 응답 비정상
- DB 페이지 저장 실패
- 전체 페이지 수집 미완료

따라서 일부 페이지만 수집된 상태에서 기존 공고가 대량 삭제되는 위험을 줄였다.

### 3.3 상세수집 제한

상세수집은 1회 실행당 최대 500건으로 제한했다.

```java
private static final int MAX_DETAIL_PER_SYNC = 500;
```

### 3.4 API 타임아웃 추가

고용24 API 호출이 무기한 대기하지 않도록 타임아웃을 설정했다.

```text
connect timeout: 10초
read timeout: 30초
```

## 4. 배포 후 정상 적용 확인

수정 전 로그:

```text
[Scheduler] API 수집 완료: 57677건
[Scheduler] DB UPSERT 완료: 57677건
```

수정 후 정상 로그:

```text
[Scheduler] 페이지 저장 완료 page=1, count=100, savedTotal=100
[Scheduler] 페이지 저장 완료 page=2, count=100, savedTotal=200
...
[Scheduler] 페이지 단위 동기화 결과: total=..., pages=.../..., fetched=..., saved=..., complete=true
[Scheduler] 마감 공고 삭제: ...건
[상세수집] 완료 — 성공: ...건, 실패: ...건
[Scheduler] 채용공고 동기화 완료
```

`페이지 저장 완료`가 반복 출력되면 페이지 단위 저장 코드가 적용된 것이다.

## 5. Ubuntu 로그 확인

Tomcat 로그 파일을 직접 보는 경우:

```bash
tail -f /path/to/catalina.out
```

systemd 서비스로 실행 중인 경우:

```bash
journalctl -u tomcat -f
```

서비스명이 `tomcat9`, `tomcat10` 등이라면 실제 서비스명으로 변경한다.

## 6. JVM 메모리 확인

Tomcat PID 확인:

```bash
PID=$(pgrep -f 'org.apache.catalina.startup.Bootstrap|tomcat')
echo $PID
```

프로세스 메모리 확인:

```bash
watch -n 5 "ps -o pid,%mem,rss,vsz,etime,cmd -p $PID"
```

JDK 도구 사용 가능 시:

```bash
jstat -gcutil $PID 5000
```

정상 기준:

- 스케줄러 실행 중 메모리는 일시 증가할 수 있음
- 페이지 저장이 진행되면서 메모리가 끝없이 증가하면 안 됨
- `OutOfMemoryError`가 재발하지 않아야 함

## 7. JVM 옵션 권장

서버 RAM이 2GB 이상인 경우 예시:

```bash
export CATALINA_OPTS="$CATALINA_OPTS -Xms512m -Xmx1024m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/jobmoa"
```

서버 RAM이 1GB 수준이면:

```bash
export CATALINA_OPTS="$CATALINA_OPTS -Xms256m -Xmx512m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/jobmoa"
```

Heap dump 경로 권한 설정:

```bash
sudo mkdir -p /var/log/jobmoa
sudo chown -R tomcat:tomcat /var/log/jobmoa
```

## 8. DB 확인 SQL

전체 채용공고 수, 상세수집 대기 건수, 최신 동기화 시간 확인:

```sql
SELECT COUNT(*) AS total,
       SUM(CASE WHEN detail_fetched = 0 THEN 1 ELSE 0 END) AS detail_pending,
       MAX(sync_dtm) AS latest_sync
FROM [J_참여자관리_JOB_POSTING];
```

특정 스케줄 실행 시각 기준 저장 건수 확인:

```sql
SELECT COUNT(*) AS synced_count
FROM [J_참여자관리_JOB_POSTING]
WHERE sync_dtm = '2026-05-20 18:00:00';
```

`sync_dtm` 값은 실제 스케줄러 로그에 출력된 값으로 변경해서 사용한다.

## 9. 실패 판정 기준

아래 로그가 나오면 정상 완료가 아니다.

```text
java.lang.OutOfMemoryError: Java heap space
[Scheduler] 전체 페이지 수집 미완료
[Scheduler] 페이지 단위 수집 실패
[Scheduler] 페이지 저장 실패
```

단, `전체 페이지 수집 미완료` 또는 `페이지 저장 실패`가 발생한 경우에도 기존 공고 삭제와 상세수집은 생략된다.

## 11. 운영 체크리스트

- [ ] 배포된 WAR가 최신 코드인지 확인
- [ ] 스케줄러 로그에 `페이지 저장 완료`가 반복 출력되는지 확인
- [ ] `페이지 단위 동기화 결과` 로그에서 `complete=true` 확인
- [ ] `OutOfMemoryError` 재발 여부 확인
- [ ] Tomcat/JVM 프로세스 메모리가 계속 증가하지 않는지 확인
- [ ] `J_참여자관리_JOB_POSTING`의 `sync_dtm` 최신값 확인
- [ ] AI 채용 추천 기능에서 채용공고 조회가 정상 동작하는지 확인

## 12. 결론

이번 수정은 기존 OOM의 핵심 원인으로 보이는 대량 채용공고 전체 메모리 적재를 제거한 변경이다.

수정 후에도 JVM 힙 크기가 지나치게 작거나 다른 기능의 메모리 누수가 있으면 OOM 가능성은 남아 있다. 따라서 코드 배포와 함께 JVM `-Xmx`, Heap dump 옵션, 배포 후 로그 모니터링을 같이 적용해야 한다.
