## 3. Task 7: Kakao SDK 초기화 실패 [차우선순위]

### 3.1 현황 분석

**오류 메시지:**
```
participant.login:158 Uncaught ReferenceError: Kakao is not defined
    at participant.login:158:9
```

**원인:** `src/main/webapp/WEB-INF/views/participantMain.jsp` (line 148-154):
```html
<!-- Kakao JavaScript SDK (카카오톡 공유 기능) -->
<script defer src="https://t1.kakaocdn.net/kakao_js_sdk/2.7.4/kakao.min.js"
        integrity="sha384-DKYJZ8NLiK8MN4/C5P2dtSmLQ4KwPaoqAfyA/DfmEc1VDxu4yyC7wy6K1Hs90nka"
        crossorigin="anonymous"></script>
<script>
    Kakao.init('${kakaoJsKey}');   ← line 153: SDK 로드 전 즉시 실행됨
</script>
```

**근본 원인:**
1. Kakao SDK 스크립트에 `defer` 속성 → DOM 파싱 완료 후 실행
2. `Kakao.init()` 인라인 스크립트 → `defer` 불가, DOM 파싱 중 즉시 실행
3. 인라인 스크립트 실행 시점에 SDK가 아직 로드되지 않아 `Kakao is not defined` 발생

**참고:** 이 문제는 이전 LCP 성능 최적화 (2026-05-19) 작업에서 Kakao SDK에 `defer`를 추가하면서 발생한 부작용이다. LCP 개선 당시 인라인 `Kakao.init()` 호출이 defer와 충돌하는 것을 감지하지 못한 것으로 판단된다.

### 3.2 수정 방안

**권장: defer 유지 + DOMContentLoaded 래핑**

LCP 성능 최적화에서 적용한 `defer`를 유지하면서 초기화 타이밍만 수정하는 것이 성능과 안정성 모두에서 유리하다.

| 파일 | 라인 | 변경내용 |
|------|------|----------|
| `src/main/webapp/WEB-INF/views/participantMain.jsp` | 149-154 | defer 유지 + init 코드를 DOMContentLoaded 래핑 |

**변경 전:**
```html
<script defer src="https://t1.kakaocdn.net/kakao_js_sdk/2.7.4/kakao.min.js"
        integrity="sha384-..." crossorigin="anonymous"></script>
<script>
    Kakao.init('${kakaoJsKey}');
</script>
```

**변경 후:**
```html
<script defer src="https://t1.kakaocdn.net/kakao_js_sdk/2.7.4/kakao.min.js"
        integrity="sha384-..." crossorigin="anonymous"></script>
<script>
    document.addEventListener('DOMContentLoaded', function() {
        if (typeof Kakao !== 'undefined' && !Kakao.isInitialized()) Kakao.init('${kakaoJsKey}');
    });
</script>
```

- `defer` 유지 → LCP 성능 최적화 효과 보존 (DOM 파싱 차단 방지)
- `DOMContentLoaded`는 모든 defer 스크립트 실행 후 발화 → SDK 로드 완료 보장
- `typeof Kakao !== 'undefined' && !Kakao.isInitialized()` 이중 방어: CDN 장애 시 에러 방지 + 중복 초기화 방지
- 이 패턴은 2026-05-19 LCP 최적화에서 Sortable 등 다른 라이브러리에 동일하게 적용한 방식과 일관됨

### 3.3 영향 범위

- `recommend-modal_0.0.3.js`의 Kakao 공유 기능 (`sendKakaoShare()`, `shareSelectedViaKakao()`)에 직접 영향
- SDK 초기화 실패 시 모든 Kakao 공유 기능이 동작하지 않음
- Task 4-4 (Kakao 공유 시 마감 알림)의 **선행 조건**

### 3.4 테스트 체크리스트

- [ ] 페이지 로드 시 브라우저 콘솔에 `Kakao is not defined` 에러 없음

### 6.2 세부과제 4-2: 삭제 정책 변경 (sync 기반 → 6개월 기준)

**현재 상태:**

`src/main/resources/mappings/Recruitment-mapping.xml` (line 291-294):

`src/main/java/com/jobmoa/app/recruitmentFormation/biz/RecruitmentScheduler.java`:
- 크론: `0 0 0,6,12,18 * * *` (매일 4회 실행)
- Phase 1: work24 API에서 전체 공고 동기화 (MERGE/UPSERT)
- 이후: `deleteOldPostings(syncDtm)` 호출 → 현재 동기화에서 갱신되지 않은 공고 = API에서 사라진 공고 → **즉시 하드 삭제**

**문제점:**
- API에서 일시적으로 누락된 공고도 삭제됨 (임시 장애 등)
- 삭제된 공고에 대한 AI 추천 이력 참조 불가
- `is_active` 컬럼이 존재하지만 삭제/필터링에 활용되지 않음

**변경 방안 (2단계 처리 — soft delete 후 hard delete):**

단순히 `deleteOldPostings`를 `created_at < 6개월`로 바꾸면, API에서 사라진 공고가 6개월 동안 `is_active = 1`로 남아 AI 추천에 계속 포함되는 문제가 생긴다. 따라서 **비활성화(soft delete) → 삭제(hard delete)** 2단계 흐름으로 변경해야 한다.

**Step 1 — 비활성화: API에서 사라진 공고를 `is_active = 0`으로 전환 (필수)**
```xml
<!-- 신규 쿼리: deactivateOldPostings -->
<update id="deactivateOldPostings" parameterType="String">
    UPDATE J_참여자관리_JOB_POSTING
    SET is_active = 0
    WHERE sync_dtm <![CDATA[<]]> #{syncDtm}
      AND is_active = 1
</update>
```
- 매 동기화 후 호출: sync_dtm이 현재 동기화 시각보다 오래된 공고 = API에서 사라진 공고 → `is_active = 0`
- API에서 재등장한 공고는 MERGE/UPSERT 시 `sync_dtm`이 갱신되므로 비활성화 대상에서 자동 제외됨

**Step 1-1 — MERGE UPDATE 시 `is_active = 1` 복구 (필수)**
```xml
<!-- 기존 upsertPosting의 WHEN MATCHED 절에 추가 -->
WHEN MATCHED THEN UPDATE SET
    ...기존 필드들...,
    is_active = 1   /* 재등장 공고 복구 */
```
- 이전에 `is_active = 0`이 된 공고가 다시 API에 나타나면 자동으로 활성 복구

**Step 2 — 삭제: 비활성화 후 6개월 이상 경과한 공고만 하드 삭제**

`created_at` 기준으로 삭제하면 오래전에 등록된 공고가 오늘 비활성화되어도 즉시 삭제되는 문제가 있다. "비활성화 후 6개월 보존"이 목적이므로 **비활성화 시점 기록이 필요하다.**

**DDL — `deactivated_at` 컬럼 추가:**
```sql
ALTER TABLE J_참여자관리_JOB_POSTING
ADD deactivated_at DATETIME NULL;
```

**Step 1의 deactivateOldPostings 쿼리에도 반영:**
```xml
<update id="deactivateOldPostings" parameterType="String">
    UPDATE J_참여자관리_JOB_POSTING
```

### 변경 1: 채용공고 스케줄러 OOM 대응 — 페이지 단위 동기화
