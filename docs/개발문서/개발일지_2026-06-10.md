## 1. 배경 및 목적

프로젝트 보완 작업을 점검한 결과, 메모리에 "미구현"으로 추적되던 항목 중 일부가 실제로는 **버그**였음을 확인했고, 대시보드–참여자 조회 간 연동과 코드 품질 정리가 필요했다.

- **상담일정 알림:** 인프라(스케줄러·WebSocket·클라이언트 구독·UI)가 양끝 모두 구축됐으나, 발행/구독 토픽 경로 불일치로 알림이 전혀 전달되지 않았다.
- **금일업무 드릴다운:** 대시보드 "금일 업무" 4개 지표를 클릭해 해당 조건 참여자를 곧바로 조회하는 동선이 없었다.
- **페이지네이션 필터 누락:** 위 드릴다운 도입 후, 딥링크로 진입한 참여자 조회에서 페이지를 넘기면 필터가 사라지는 버그가 드러났다.
- **코드 품질:** 운영 콘솔에 남은 디버그 로그와 조용히 삼켜지는 예외 정리가 필요했다.

## 2. 변경 내용 (코드 리뷰)

### 2-1. 상담일정 알림 토픽 불일치 버그 수정 — `ScheduleAlertScheduler.java`

스케줄러가 `sendObjectToUser(...)`(→ `messagingTemplate.convertAndSendToUser` → `/user/{id}/topic/schedule-alert`)로 발행했으나, 클라이언트(`gnb-notification_0.0.2.js`)는 일반 토픽 `/topic/schedule-alert/{userId}`를 구독하여 **경로가 일치하지 않아 미전달**되었다. 정상 동작하던 형제 알림(`schedule-modified`, `resume-request`)과 동일한 일반 토픽 패턴으로 통일.

```java
// 변경 전
webSocketService.sendObjectToUser("/topic/schedule-alert", notification, target.getCounselorId());
// 변경 후 (WebSocketService.sendObject → convertAndSend 재사용)
webSocketService.sendObject("/topic/schedule-alert/" + target.getCounselorId(), notification);
```

- JavaDoc 토픽 설명을 `/topic/schedule-alert/{상담사ID}`로 갱신.
- 스케줄러 `catch`의 로그를 `e.getMessage()`만 → 스택트레이스 포함(`log.error(msg, e)`)으로 보강.

### 2-2. 대시보드 "금일 업무" 지표별 참여자 조회 드릴다운 — `DashBoardPage.jsp`, `dashboard_0.0.4.css`

"금일 업무" 위젯 4개 지표가 참여자 조회 화면의 기존 `searchTypeList` 필터와 1:1로 대응하고, 참여자 조회(`/participant.login`)가 GET 쿼리스트링 딥링크를 이미 지원함을 확인 → **백엔드/매퍼 변경 없이** 각 지표를 클릭 가능한 앵커로 변환.

| 금일업무 지표 | 딥링크 필터값(`searchTypeList`) |
|---|---|
| 초기상담 미실시자 | `noInitial` |
| 최근상담 21일 경과 | `recent21` |
| 구직 만료 도래 | `jobExpire` |
| 기간 만료 예정 | `periodExpire` |

```html
<a href="/participant.login?searchTypeList=noInitial&endDateOptionList=false"
   class="work-list-item work-list-link text-decoration-none text-reset" title="해당 참여자 조회로 이동"> ... </a>
```

- `endDateOptionList=false`로 진행중만 필터.
- `.work-list-link` 호버 affordance CSS 추가(`dashboard_0.0.3.css` → `0.0.4.css` 버전 bump).

### 2-3. 페이지네이션 필터 누락 버그 수정 — `paginationJS_0.0.2.js`

공용 페이지네이션 유틸 `searchHref()`가 **현재 쿼리스트링의 첫 파라미터를 무조건 `page=`로 간주해 버리는** 구조였다(`search.indexOf(item) > 0`). 폼 제출 시에는 hidden `page=1`이 첫 번째라 우연히 동작했으나, 금일업무 딥링크(`?searchTypeList=noInitial&...`, page 없음)로 진입하면 첫 파라미터 `searchTypeList`가 페이지 이동 시 누락되었다. 추가로 `window.location.search`의 선행 `?`가 첫 항목을 손상시켰다.

```js
// 변경 전: 첫 파라미터 강제 제외 + '?' 손상 + 다중값 취약
let search = window.location.search.split('&');
search.forEach(item => { if (search.indexOf(item) > 0) href += '&' + item });

// 변경 후: URLSearchParams로 순서·다중값 무관하게 page만 교체, 나머지 필터 전체 보존
const params = new URLSearchParams(window.location.search);
params.set('page', page);
return '?' + params.toString();
```

- 공용 유틸이므로 이를 참조하는 **6개 페이지 전부**에 적용(참여자조회/지점참여자/알선기업목록·상세/스타벅스 참여자목록/외부 참여자관리). 6개 JSP 참조를 `0.0.2`로 갱신.
- 정렬/행 링크 함수(`searchMainHref`, `sortHref`)는 이미 `indexOf >= 0` + `?` 제거 패턴이라 정상 → 변경 불필요.
