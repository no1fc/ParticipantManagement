## 1. 배경 및 목적

참여자 관리 페이지(`UpdateParticipantsPage.jsp`)에서 희망직무(wishJobList) 데이터가 DB에 정상 존재하고 백엔드 DTO까지 정상 로드되지만, 프론트엔드에 표시되지 않는 버그가 보고되었다. 동시에 AI 추천 시스템의 Gemini 2단계 응답 파싱 안정성을 강화하는 디버그 로그를 추가했다.

## 2. 희망직무 미표시 버그 수정

### 2-1. 근본 원인 분석

**`defer` 스크립트와 인라인 `$(document).ready()` 콜백의 실행 순서 충돌**

| 단계 | 이벤트 | 설명 |
|------|--------|------|
| 1 | jQuery 동기 로딩 | `<head>`에서 `defer` 없이 로딩 (즉시 사용 가능) |
| 2 | HTML 파싱 중 | 인라인 스크립트의 `$(document).ready()` → **콜백 A 등록** (`initWishJobList` 호출) |
| 3 | HTML 파싱 완료 | `defer` 스크립트 실행 → `jobWishListManager_0.0.1.js` 내부 `$(function(){})` → **콜백 B 등록** (`initWishJobList` 정의) |
| 4 | DOMContentLoaded | jQuery가 등록 순서대로 실행: **A → B** |
| 5 | **콜백 A 실행** | `typeof initWishJobList === 'function'` → **false** (아직 미정의) → 호출 안됨 |
| 6 | 콜백 B 실행 | `window.initWishJobList` 정의 (이미 늦음) |

### 2-2. 수정 내용

희망직무 초기화 코드를 `$(document).ready()` 블록에서 분리하여 `window.addEventListener('load', ...)` 패턴으로 변경. `window load`는 모든 defer 스크립트 실행 + DOMContentLoaded + 모든 ready 콜백 완료 후 발생하므로 `initWishJobList`가 확실히 정의된 상태에서 호출됨.

같은 JSP 내 키워드 초기화 코드(line 399)도 이미 동일한 `window.addEventListener('load', ...)` 패턴을 사용 중이므로 일관성 있는 수정.

### 2-3. 변경 파일

| 파일 | 변경내용 |
|------|----------|
| `UpdateParticipantsPage.jsp` | 희망직무 초기화를 `$(document).ready()` → `window.addEventListener('load', ...)` 로 분리 |

## 3. AI 추천 디버그 로그 강화

### 3-1. 변경 내용

| 파일 | 변경내용 |
|------|----------|
| `GeminiApiService.java` | Structured Output 스키마에 `required` 필드 추가 (`구인인증번호`, `score`, `reason`, `bestGujinNo`, `scores`), 2단계 응답 파싱에 상세 디버그 로그 추가 |

| `ParticipantJobRecommendServiceImpl.java` | scoreMap 매핑 시 null certNo 경고 로그, scoreMap 크기/키 목록 로그, certNo 미발견 경고 로그 추가 |
| `UpdateController.java` | 희망직무 조회 결과 디버그 로그 추가 (`counselJobNo`, `wishJobList`, `wishJobListSize`) |
